- Task: Unify the active widget default palette so the renderer, shared config defaults, and settings reset action no longer drift independently
- Encountered Problem: `widget_render.rs` had been changed to a different set of `DEFAULT_*` active colors while `shared-core` config defaults and the Tauri settings reset palette still used `#34C759 / #FFCC00 / #FF3B30`. That meant the same conceptual "default active color system" existed in multiple places with conflicting values.
- Thought Process: I traced the render path and config precedence. The renderer reads from `config.widget_visual.palette`, but it also performs legacy-default normalization, so its internal `DEFAULT_*` values still matter for existing configs that contain the old legacy defaults. With those defaults out of sync, behavior depended on which layer happened to supply the color.
- Options Considered: Keep multiple default sets and document the precedence; move all defaults into only one runtime layer; restore the renderer `DEFAULT_*` colors so they match shared-core and the settings reset palette while keeping the current legacy-normalization behavior.
- Chosen Solution: Change `widget_render.rs` `DEFAULT_GREEN`, `DEFAULT_YELLOW`, and `DEFAULT_RED` back to the same values used by `shared-core` and `App.tsx`, and add an inline note that these three locations must stay aligned.
- Rationale: The user asked for one unified configuration model, not multiple competing defaults for the same active lamp colors. Aligning the renderer defaults with the existing shared config and UI reset palette removes an avoidable source of drift without expanding scope.
