- Task: TSM-VAL-16 解决 `request_refresh` 经由 named pipe / Tauri 调用时出现的 `拒绝访问 (os error 5)`
- Encountered Problem: `request_refresh` 在 live host 上返回 `SettingsIpcResponse::Error`，错误信息为 `拒绝访问。 (os error 5)`，而 `get_snapshot` / `get_settings` 同时是正常的。
- Thought Process: 我先把反馈环收窄到单命令验证，再对照 `request_manual_refresh_command()` 的调用栈，确认失败点只可能出在“记录 `last_manual_refresh_at` 并持久化配置”这一步，而不是 detector 或 Tauri 前端。
- Options Considered: 1. 继续追文件占用来源。2. 给配置写入换成更复杂的重试/原子写。3. 把 `last_manual_refresh_at` 改成运行时内存字段，不让 refresh 热路径依赖配置落盘。
- Chosen Solution: `request_manual_refresh_command()` 先向主窗口投递刷新命令，再仅更新内存中的 `SETTINGS_CONFIG.diagnostics.last_manual_refresh_at`，不再把这个诊断时间戳写回 `config.json`。
- Rationale: 手动刷新时间戳属于运行时诊断，不是必须持久化的用户配置。把它从热路径持久化中拿掉后，`request_refresh` 可以稳定成功，同时前端仍能通过轮询 `get_settings` 看到最新时间戳。
