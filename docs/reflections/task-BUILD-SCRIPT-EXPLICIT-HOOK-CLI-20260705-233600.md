- Task: Make the workspace root build script explicitly rebuild the hook CLI before rebuilding the Win32 host
- Encountered Problem: The repository root `pnpm build` already triggered `cargo build -p taskbar-widget --offline`, but the hook CLI remained an implicit side effect of the host package build. That made it too easy to miss when the active hook config was pointing at a stale binary, especially after the workspace target-path drift issue.
- Thought Process: I checked the existing root build chain and confirmed it already covered frontend assets, the standalone Tauri settings binary, and the Win32 host. The gap was not capability but visibility: the hook writer is operationally critical, yet it was not named as a first-class build artifact.
- Options Considered: Leave the script unchanged and rely on docs; add only a post-build timestamp print; insert an explicit `cargo build -p taskbar-widget --bin taskbar_widget_hook --offline` step and print artifact metadata for all three runtime binaries.
- Chosen Solution: Update `scripts/rebuild-all.ps1` to build the hook CLI explicitly before the host build, then print the resolved root `target\debug` paths plus timestamps and sizes for the hook CLI, host, and Tauri settings executable.
- Rationale: This turns the hook state-writer from an implicit byproduct into an explicit build contract, makes path drift easier to spot immediately, and still preserves the repo rule that the final host acceptance build remains a separate `cargo build -p taskbar-widget --offline` step.
