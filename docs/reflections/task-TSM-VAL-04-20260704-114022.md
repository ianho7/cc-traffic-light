- Task: TSM-VAL-04 Tauri settings lifecycle loader blocker
- Encountered Problem: `taskbar-widget.exe` 在 `main()` 前报 `TaskDialogIndirect` 入口缺失，阻断 settings lifecycle 验证；后续脚本还把无 taskbar 环境下的 tray 添加失败和强杀宿主误判为生命周期失败。
- Thought Process: 先构建只静态导入 `TaskDialogIndirect` 且嵌 Common Controls v6 manifest 的最小 probe，确认当前机器/SxS/manifest 机制可用。再反查导入来源，发现 workspace 构建中 Tauri 的 `common-controls-v6` feature 会启用同一 `muda` 依赖的 `TaskDialogIndirect` 路径，并污染宿主产物。随后循环修正验证脚本：分包构建、托盘失败非致命、按 PID 查 Tauri 窗口、用 `WM_CLOSE` 正常关闭宿主。
- Options Considered: 继续修改 manifest；patch `muda` 或 Slint；只要求人工运行 `cargo build -p taskbar-widget`；让 lifecycle 脚本固定分包构建并验证最终 exe。
- Chosen Solution: lifecycle 脚本先构建 `taskbar-settings-tauri`，最后单独构建 `taskbar-widget`，避免 workspace feature 统一污染宿主；宿主在 `Shell_NotifyIconW(NIM_ADD)` 失败时记录日志并继续运行；脚本用窗口 PID fallback 和正常 `WM_CLOSE` 验证 lifecycle。
- Rationale: 这保留当前 Tauri/Slint 依赖结构，不引入依赖 patch，同时让自动验证产物稳定来自正确 feature set。最终 `validate-tauri-settings-lifecycle.ps1 -Configuration debug -TimeoutSeconds 20` 在非 sandbox 桌面环境通过，所有 lifecycle 布尔项为 `true`。
