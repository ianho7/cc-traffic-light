- Task: TSM-VAL-04 debug-launch settings window validation fix
- Encountered Problem: Running `.\target\debug\taskbar-widget.exe` and opening settings from the tray launched the Tauri window, but the WebView navigated to `http://127.0.0.1:1420` and showed a refused-connection page instead of loading bundled frontend assets.
- Thought Process: I first separated process-launch failure from frontend-load failure by checking that the settings process was successfully spawned and that the symptom was a WebView navigation error. I then traced Tauri's build-mode selection through the generated Cargo build output and the local crate sources to identify why a plain debug build still behaved like a dev-server build.
- Options Considered: Leave `devUrl` empty and give up on dev-server workflow; change the host to launch a different packaged executable path; enable Tauri's production-mode feature so ordinary `cargo build` outputs embed or resolve local assets instead of navigating to `devUrl`.
- Chosen Solution: Enable `custom-protocol` as a default feature in `taskbar-settings-tauri/src-tauri/Cargo.toml` and add a static verification check for that invariant in `taskbar-settings-tauri/scripts/verify-settings-coverage.ps1`.
- Rationale: Local Tauri sources show that `cfg(dev)` is derived from `not(feature = "custom-protocol")`, and `get_app_url()` uses `devUrl` whenever `cfg(dev)` is active. Making `custom-protocol` the default preserves the current `cargo build -p taskbar-settings-tauri --offline` validation flow, removes the `127.0.0.1:1420` dependency for the host-launched debug executable, and gives a small, auditable regression guard in the existing verification script.
