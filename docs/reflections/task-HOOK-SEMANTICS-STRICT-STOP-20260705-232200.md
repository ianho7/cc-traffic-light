- Task: Align Codex/Claude hook state rendering with explicit hook semantics instead of Stop-payload error heuristics
- Encountered Problem: The taskbar settings overview could show `error` for Codex after an otherwise normal turn end because `hook_rules.rs` treated `Stop` as `error` whenever any payload text happened to contain broad failure words.
- Thought Process: I compared the current implementation against the repository's own lifecycle-hook checklists and validation notes. Those documents consistently describe explicit failure hooks such as `StopFailure` and `PostToolUseFailure` as the error path, while `Stop` is the normal end-of-turn event. The implementation had drifted into a heuristic that overrode hook semantics with payload text.
- Options Considered: Keep the heuristic and narrow the keyword list; keep `Stop` special-cased based on previous waiting state; switch `Stop` back to a strict semantic mapping and reserve `error` only for explicit failure hooks.
- Chosen Solution: Remove payload-text error inference from `Stop`, map `Stop` and `SubagentStop` directly to `done`, and add unit tests proving that `Stop` stays `done` even when the payload text contains words like `failed` or `permission denied`.
- Rationale: The user-facing state should reflect hook semantics first. A normal `Stop` event ending a turn is not an error, and treating arbitrary payload text as a failure signal creates false alarms that undermine trust in the widget.
