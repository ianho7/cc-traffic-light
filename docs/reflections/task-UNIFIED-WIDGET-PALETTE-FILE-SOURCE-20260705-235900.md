- Task: Unify widget color management around a single file-backed palette model shared by the Win32 widget and the Tauri settings shell
- Encountered Problem: The active lamp defaults, inactive lamp rendering, and Tauri reset behavior were split across multiple layers. The widget renderer still hardcoded per-lamp inactive colors, the Tauri UI carried its own reset defaults, and first launch did not guarantee a persisted default config file for the palette.
- Thought Process: I traced the full path from `%APPDATA%\CcTrafficLight\config.json` through `shared-core`, host IPC, widget rendering, and the Tauri appearance page. The cleanest fix was to make `shared-core::app_config` the palette authority, persist defaults on missing config, let both widget and Tauri reload from that file-backed model, and reduce inactive styling to one derived brightness parameter instead of multiple separate colors.
- Options Considered: Keep the existing `off` color and only brighten the renderer; move all palette defaults into Tauri and push them down to Rust; keep active colors in config but continue hardcoding inactive colors in the renderer; collapse the model to three active colors plus one shared inactive-brightness control.
- Chosen Solution: Bump the config schema, replace the old `off` palette entry with `inactive_brightness_percent`, auto-create the default config file on first load, expose the authoritative default widget palette through the Tauri bootstrap DTO for reset actions, refresh the host config cache from disk, and derive inactive lamp shells from the active hue using one shared brightness multiplier.
- Rationale: This matches the requested storage model more closely: one persisted palette source, one reset source, one inactive-derivation control, and no separate hidden per-lamp inactive color table that can drift away from the config file.
