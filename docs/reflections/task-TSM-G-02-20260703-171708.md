- Task: TSM-G-02 设置修改先写回 shared-core，再由宿主应用
- Encountered Problem: Tauri 写路径虽然已经能把完整 `AppConfig` 送回宿主，但宿主侧保存仍直接调用 `app_config::save_config`，shared-core 的 settings service 边界没有真正承担保存职责。
- Thought Process: 我不新增新的 façade，而是直接复用已经存在的 `HostSettingsBridge: SettingsService` 实现，把当前所有宿主保存入口都切到 `save_settings()`，再保留宿主侧对 autostart 等平台副作用的应用职责。
- Options Considered: 保持直接写 `app_config::save_config`；重写一层新的 shared-core settings façade；让宿主已有 bridge 正式通过 `SettingsService` 承担保存职责。
- Chosen Solution: 更新 `taskbar-widget/src/settings_bridge.rs`，让 `update_config`、`toggle_autostart_setting` 和 `apply_full_settings` 都通过 `HostSettingsBridge::save_settings()` 落盘，再由宿主继续处理真正的平台应用动作。
- Rationale: 这样可以在不扩大架构面的前提下，把写路径正式收回 shared-core service 边界，同时保留宿主对 Win32 / 注册表副作用的控制权。
