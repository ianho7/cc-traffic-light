- Task: Align widget lamp rendering with the root README state contract and make all three lamp positions persist visibly in single-source mode
- Encountered Problem: In the real taskbar widget, `working` and `completed` could appear to "lose" one lamp position or leave the active slot looking empty, while the root README now specifies that every state should keep a stable three-lamp layout with only the active lamp changing color or animation.
- Thought Process: I compared the README contract, the actual single-group taskbar width, and the renderer tests. The earlier tests only exercised a 160px frame, while the real single-group layout uses an 80px module. After adding 80px single-group renderer tests, I traced the remaining failure to the active overlay alpha math: the renderer used `u8::saturating_mul()` before dividing by 255, so the intended colored overlay collapsed to an alpha near `1` and looked effectively invisible.
- Options Considered: Treat the issue as a stale executable problem only; keep the renderer as-is and just change state mapping; make the renderer always draw the gray shell for every lamp position and then overlay the active colored lamp on top.
- Chosen Solution: Re-align `completed` with the root README's left-green contract, change `widget_render.rs` to always draw off-state shells before any active overlay, fix the overlay alpha scaling with a dedicated `scale_alpha()` helper that performs the multiply in a wider integer type, and add 80px single-group renderer tests for idle, working, and completed.
- Rationale: This preserves the visual invariant the user expects: three stable lamp slots are always present, and state changes only affect color and animation. It also fixes the actual rendering defect instead of just shifting state mappings or blaming stale binaries, and it makes the tests match the real taskbar width instead of an easier synthetic frame.
