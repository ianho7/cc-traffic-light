- Task: TSM-VAL-11 Tauri 异常退出后，tray 再次打开 settings 能恢复
- Encountered Problem: 仅靠一次正常打开/关闭无法覆盖“settings 崩掉或被杀后，宿主是否还能恢复”的路径，而这正是从 Slint 切主入口到 Tauri 后必须补证的一项稳定性门槛。
- Thought Process: 在脚本里保留强杀 settings 进程的步骤，但不再强杀宿主。这样可以单独验证“异常退出的 settings”与“正常关闭的宿主”这两种不同生命周期语义。
- Options Considered: 不做异常退出验证；只手工结束进程；在脚本中强杀 settings 后再次发送 Open Settings，并检查新 PID 产生。
- Chosen Solution: 用 `Stop-Process -Id $report.second_settings_pid -Force` 模拟 Tauri 异常退出，再验证 `reopen_after_forced_kill_spawned = true` 和新的 `third_settings_pid` 被记录。
- Rationale: 这正好覆盖了用户最关心的恢复路径，也能从自动 report 中持续回归。当前最新 report 已经证明该恢复路径通过。
