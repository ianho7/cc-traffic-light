- Task: TSM-VAL-05 从 tray 打开 settings 时，默认进入 Tauri 路径
- Encountered Problem: 之前用户手工日志里只出现了 `slint-settings show result=true`，没有任何 `settings-process spawned...`，说明实际运行过旧产物或被环境变量强制进了 Slint，无法证明默认入口已经是 Tauri。
- Thought Process: 先验证脚本是否使用根 workspace 的 [target/debug/taskbar-widget.exe](/D:/project/cc-traffic-light/target/debug/taskbar-widget.exe)，再在脚本里强制 `CC_TRAFFIC_LIGHT_SETTINGS_HOST=tauri` 和 `CC_TRAFFIC_LIGHT_TAURI_SETTINGS_EXE=<settings exe>`，最后用 runtime log 确认打开动作走的是 `settings_process::open_or_focus_tauri_settings()`。
- Options Considered: 只看 UI 外观；只看 settings 进程是否存在；结合 runtime log、report 和脚本构建路径一起证明默认进入 Tauri。
- Chosen Solution: 以修正后的 lifecycle 脚本和 runtime log 作为主证据，确认每次 Open Settings 都记录 `spawned tauri settings pid=...` 或 `reused existing tauri settings pid=...`，且无 `falling back to slint`。
- Rationale: 这比人工观察更可靠，也排除了运行旧 exe 的干扰。当前 report 和 runtime log 已经证明 tray/open 命令进入的是 Tauri 主路径。
