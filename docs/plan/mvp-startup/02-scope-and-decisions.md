# 02. 范围与关键决策

## 1. 这个 MVP 到底是什么

这是一个技术验证型 MVP，不是产品 MVP。

它不是为了给用户用。

它是为了回答一个明确问题：

> Rust 能不能在 Windows 任务栏里稳定嵌入一个自定义模块？

## 2. 一定要守住的范围

### 只做一个模块

模块内容固定，例如：

- `TASKBAR WIDGET`
- `HELLO TASKBAR`

不要在 MVP 阶段引入：

- 多模块
- 可配置模块
- 模块管理器

### 只做一个系统路径

MVP 只应该支持：

- 一台机器
- 一个 Windows 版本
- 一个任务栏结构

如果当前验证环境是 Win11，就只做 Win11 最小路径。  
如果当前验证环境是 Win10，就只做 Win10/经典任务栏最小路径。

不要同时做两套。

### 只做最小绘制

绘制内容只需要：

- 固定背景色
- 一段文本

可选但不是必须：

- 细边框，方便确认模块边界

不要做：

- 图标
- 透明
- 渐变
- 动画
- 图表

## 3. 必须采用的技术决策

### 决策 1：使用 `windows` crate

原因：

- 直接访问 Win32 API
- 不需要引入额外 GUI 框架
- 最适合任务栏嵌入这种底层行为

### 决策 2：自己管理消息循环

原因：

- 这是 Win32 宿主窗口的核心
- 后续 `WM_PAINT`、`WM_DESTROY`、`WM_TIMER` 都要依赖它

### 决策 3：窗口过程里只保留最少逻辑

原因：

- Rust + Win32 回调本身就有 `unsafe`
- 如果把复杂状态和逻辑堆进去，调试会很痛苦

建议：

- 回调里只分发消息
- 真正逻辑放到小函数

### 决策 4：先做 GDI，不做 D2D

原因：

- 这是最稳的首条路径
- 有利于先排查宿主嵌入和定位问题

### 决策 5：允许保守定位

原因：

- MVP 不需要最优布局
- 只要“稳定可见且像任务栏内模块”就够了

## 4. 当前不允许引入的抽象

以下抽象在 MVP 阶段都不应该出现：

- 通用 `Widget` trait
- 插件 ABI
- 多宿主策略框架
- 配置驱动布局系统
- 主题系统
- 跨平台抽象层

唯一可以接受的抽象是：

- 一个最小 `taskbar` 模块
- 一个最小 `win32` 工具模块

## 5. 推荐的最小行为模型

程序启动后：

1. 注册窗口类
2. 创建窗口
3. 查找任务栏宿主
4. 调 `SetParent`
5. 计算位置
6. 调 `MoveWindow`
7. 进入消息循环
8. `WM_PAINT` 里绘制固定文本

就这些。

## 6. 哪些问题现在不要解决

以下问题如果现在开始处理，基本就是范围失控：

- 窗口重启后自动恢复任务栏状态
- Explorer 崩溃/重启后的重挂载
- 任务栏位置变动时自动自愈
- 主任务栏与副任务栏自动切换
- 多语言文本渲染
- 高 DPI 完整适配

这些都是真问题，但不是 MVP 问题。

## 7. 什么时候算应该进入下一阶段

只有满足下面全部条件，才应该做第二阶段：

- 模块能稳定显示
- 不是浮窗
- 关闭重开可重复
- 日志能说明宿主句柄和定位信息

如果这 4 条都没稳定，就不要扩范围。
