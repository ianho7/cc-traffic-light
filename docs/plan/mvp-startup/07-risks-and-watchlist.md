# 07. 风险与范围失控清单

## 1. 最大风险不是 Rust

最大风险是：

- 任务栏内部结构依赖

也就是：

- 你依赖 `Explorer` 当前版本的窗口树
- 不是稳定官方 ABI

Rust 只能帮你把内存和状态管理做得更稳。

Rust 不能消除任务栏结构变化风险。

## 2. 风险清单

### 风险 1：目标系统结构不符合预期

- Risk: 当前 Windows 版本的任务栏窗口结构和参考资料不一致
- Impact: 找不到正确父窗口，或嵌入后不可见
- Likelihood: 中到高
- MVP Mitigation: 只锁定一个目标系统，先做最小路径
- Fallback Plan: 改为从更容易调试的系统版本开始

### 风险 2：Rust Win32 回调生命周期处理不当

- Risk: 回调里对象状态和窗口句柄管理出错
- Impact: 崩溃、空指针、窗口行为异常
- Likelihood: 中
- MVP Mitigation: 只保留一个最小状态对象，不提前建复杂模型
- Fallback Plan: 先使用更直接的状态保存方案，等 MVP 成功后再整理

### 风险 3：定位逻辑错误导致窗口不可见

- Risk: 模块挂进去了，但坐标错误
- Impact: 肉眼看起来像失败
- Likelihood: 高
- MVP Mitigation: 使用醒目背景色和固定保守坐标
- Fallback Plan: 先验证窗口是否属于父宿主，再逐步修坐标

### 风险 4：过早追求 Win11 完整兼容

- Risk: 同时处理 Widgets、左对齐、居中、副屏
- Impact: 范围急剧膨胀
- Likelihood: 高
- MVP Mitigation: 明确禁止多分支并行
- Fallback Plan: 退回单路径

### 风险 5：过早追求“好架构”

- Risk: 花大量时间设计 traits、模块层、插件接口
- Impact: 一直在搭框架，没有用户可见结果
- Likelihood: 很高
- MVP Mitigation: 明确限制文件数和抽象数
- Fallback Plan: 回退到最小可运行路径

## 3. Over-Engineering Watchlist

以下事情在 MVP 阶段必须避免：

- 不要引入 GUI 框架
- 不要做插件系统
- 不要做配置系统
- 不要做多显示器
- 不要做主题系统
- 不要做 D2D / DComposition
- 不要做完整恢复机制
- 不要做 Win10/Win11 双策略框架
- 不要做通用 `Widget` 抽象
- 不要重构成“任务栏自定义平台”

## 4. 允许的最小复杂度

MVP 阶段可接受的复杂度上限：

- 1 个二进制项目
- 3 到 5 个源文件
- 1 个最小状态结构
- 1 条任务栏定位路径

超过这个规模，就该先问自己：

> 这个改动是不是直接帮助我证明 MVP 成立？

如果不是，就延期。

## 5. 进入下一阶段的前提

只有当下面都成立时，才应该继续扩展：

- 模块已可见
- 模块已嵌入任务栏
- 启停可重复
- 基本日志足够定位错误

否则不要扩范围。
