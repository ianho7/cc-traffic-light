# 素材灯组 MVP 实施计划

## 目标

交付一个最小可用的本地“素材灯组”功能：用户保存由绿、黄、红三张图片组成的灯组，并分别应用给 Codex、Claude Code；任务栏保留 Agent 标识，仅替换三盏灯的绘制，原状态映射、亮灭和闪烁规则不变。

成功信号：用户可完成“上传三图 → 裁成正方形 → 保存 → 应用至某 Agent”，并在任务栏实时看到素材灯组随原有状态规则变化。

## MVP 范围

### 必须有

- 三个固定语义槽位：绿、黄、红；三张均确认裁剪后才可保存。
- 支持 PNG、JPEG、WebP 输入，前端导出 64×64 PNG。
- 灯组名称、保存、编辑、应用、恢复默认。
- 同一灯组可同时应用给 Codex 和 Claude。
- 使用中禁止删除；素材缺失或损坏时安全回退内建灯组并显示提示。
- 保留现有 widget 大小、Agent 标识、灯位间距和整组点击打开设置的行为。
- 桌面与窄窗口均可操作。

### 不做

- 滤镜、旋转、抠图、GIF/APNG、AI 图像处理。
- 自定义灯位顺序、尺寸、间距、状态映射或单图点击行为。
- 在线素材库、导入导出、云同步、预置下载、跨设备同步。
- 重写状态检测、任务栏挂载、IPC 架构或现有灯效状态机。

## 当前基础与约束

- `crates/shared-core/src/app_config.rs` 是共享配置唯一来源，应在此扩展 schema 和兼容旧配置。
- `taskbar-widget/src/widget_render.rs` 已按三灯布局和 alpha 计算状态效果；应复用这些效果值，而非重写动画。
- `taskbar-widget/src/widget_image.rs` 已用 `image` 解码 PNG 并绘制 RGBA，适合增加本地素材缓存。
- `taskbar-settings-tauri/src/App.tsx` 已有“全量配置保存 + 通知 host”的模式。
- 当前 Tauri 后端未声明文件对话框/文件系统插件；选择文件和落盘方式需先确认现有 Tauri capability 配置，优先使用已有能力，不为 MVP 引入不必要的库。

## MVP 决策门

- 保留：本地 PNG 资产目录、配置仅存元数据/关联、复用现有渲染与 IPC。
- 简化：裁剪器用浏览器 Canvas 实现，不添加裁剪组件依赖；只输出一个固定尺寸。
- 删除：任何通用“素材资源管理框架”、图片数据库、后台扫描器。
- 延后：导入导出和可配置布局，因为它们不能验证核心换肤循环。

## 按优先级的自主执行计划

### P0：数据契约与安全回退

**目标：** 先让配置能够正确表示灯组和 Agent 绑定，即使没有 UI 或图片也绝不影响旧用户。

**涉及文件：**

- `crates/shared-core/src/app_config.rs`
- 相关 DTO/TypeScript 配置镜像：`taskbar-settings-tauri/src/types.ts`
- 共享 IPC 类型及现有配置测试

**任务：**

1. 在 `widget_visual` 下增加最小模型：`material_groups`、`codex_material_group_id`、`claude_material_group_id`。
2. `MaterialGroup` 仅含 `id`、`name`、`green_path`、`yellow_path`、`red_path`。
3. 提升 schema 版本；旧配置反序列化为无素材组、两 Agent 均使用默认灯组。
4. 在规范化阶段限制：ID 为空、槽位路径为空或重复/无效组不应导致崩溃；host 最终按默认灯组渲染。
5. 覆盖配置 round-trip、旧 schema 兼容、绑定未知 ID 时回退的 Rust 单测。

**预期结果：** 配置模型稳定、向后兼容、可安全驱动后续 host 和 UI。

**MVP 检查：** 这是跨进程数据契约和升级安全线；不包含文件写入或 UI。

### P1：Host 素材加载、缓存与渲染替换

**目标：** 通过配置即可证明“同样状态规则 + 自定义图像”的完整 host 闭环。

**涉及文件：**

- `taskbar-widget/src/widget_image.rs`
- `taskbar-widget/src/widget_render.rs`
- 必要时 `taskbar-widget/src/main.rs`
- `taskbar-widget` 单测

**任务：**

1. 增加按灯组 ID 和文件修改状态加载的 PNG 缓存；不可在每一帧重新读文件或解码。
2. 将现有单灯 16×16 位置作为素材画布区域，按已有 RGBA 混合函数绘制。
3. 复用现有 alpha：三图常驻；非激活素材使用暗态 alpha；激活素材按长亮/闪烁 alpha 变化。
4. 任一素材读取或解码失败时，整个对应 Agent 回退内建圆灯，并记录可诊断错误；不写回或删除用户素材。
5. 保持原 `hot_zones` 不变，确保自定义模式仍可点击打开设置。
6. 添加默认回归、有效素材替换、暗态、激活态、缺图回退、两 Agent 应用不同/相同灯组的测试。

**预期结果：** 现有灯效逻辑不变，自定义图片能以同一规则显示。

**MVP 检查：** 这是用户可见核心；不扩展状态机、任务栏挂载或布局。

### P2：素材落盘命令与最小文件一致性

**目标：** 设置端安全保存已裁剪素材，避免将 Base64 写进 `config.json`。

**涉及文件：**

- `taskbar-settings-tauri/src-tauri/src/*` 的现有 command/IPC 边界
- `crates/shared-core/src/app_config.rs`
- 必要时 host named-pipe 命令及 TypeScript `lib/tauri.ts`

**任务：**

1. 提供最小命令：保存一个完整灯组、删除一个未被引用灯组。
2. 目标目录固定为 `%APPDATA%\\CcTrafficLight\\assets\\<group-id>\\`，槽位文件固定为 `green.png`、`yellow.png`、`red.png`。
3. 保存采用临时文件后替换；三张都成功才更新配置，失败则保留旧配置与旧素材。
4. 限制处理后的 PNG 为 64×64；前端负责生成，后端仅验证 PNG 和合理文件大小。
5. 删除前检查两个 Agent 的绑定；被引用时返回可显示的错误。
6. 成功保存/应用后走现有 `saveSettings + notifySettingsApplied` 通道，促使 host 缓存失效并重绘。

**预期结果：** 文件与配置不会形成半完成状态。

**MVP 检查：** 仅使用本地文件和已有通信模式，不建立素材数据库或后台服务。

### P3：设置页的单一主流程

**目标：** 用户无需理解文件结构即可创建并应用灯组。

**涉及文件：**

- `taskbar-settings-tauri/src/App.tsx`
- 新增 `taskbar-settings-tauri/src/pages/MaterialLibraryPage.tsx`
- 必要的组件、CSS、Paraglide 文案
- `taskbar-settings-tauri/src/types.ts`
- `taskbar-settings-tauri/src/lib/tauri.ts`

**任务：**

1. 在现有设置导航新增“素材灯组”页。
2. 显示默认灯组与已保存灯组列表，且明确 Codex/Claude 当前绑定。
3. 创建/编辑时显示三个固定卡片：绿、黄、红。
4. 每张卡片：选择图片 → Canvas 预览 → 拖动平移、滑杆缩放 → 确认裁剪。
5. 三张皆完成且名称非空时才允许“保存灯组”。
6. 灯组卡片提供“应用到 Codex”“应用到 Claude Code”“恢复默认”“删除”；删除和覆盖素材均二次确认。
7. 读取/应用到缺失资源时展示“素材不可用，已回退默认灯组”。

**最小 UX：**

- 复用现有页面容器、按钮和 pending 状态，不建立新的设计系统。
- 宽窗口用三列素材卡片；窄窗口改为单列堆叠。
- 不做复杂拖放排序、批量操作、搜索、标签或分页。

**预期结果：** 用户可以独立完成创建、保存和应用。

**MVP 检查：** 这是唯一必要的新 UI；无关管理能力全部延后。

### P4：端到端验证与交付记录

**目标：** 提供可重复的工程证据，不把“能编译”误作功能完成。

**任务：**

1. 运行：
   - `cargo test --workspace --offline`
   - `cargo check -p taskbar-widget --offline`
   - `pnpm build`
   - 单独按顺序构建 settings 后再构建 host，避免 workspace 特征污染。
2. 在 Windows 手工验证：
   - 三图未齐不可保存；
   - 保存、应用至一个 Agent、再复用至另一 Agent；
   - 默认/暗态/常亮/闪烁状态；
   - 点击自定义素材打开设置；
   - 删除保护；
   - 手工删除素材文件后回退和提示；
   - 宽窗口、窄窗口裁剪与保存流程。
3. 更新一份 `docs/handoff/` 或 `docs/reflections/`，记录 config schema、资产路径、验证的 host exe 绝对路径，以及素材丢失回退结果。

**预期结果：** 完成分层测试和真实 Windows 桌面验证。

## 自主执行顺序

严格串行执行 `P0 → P1 → P2 → P3 → P4`。

P0 固定跨进程数据契约；P1 最早验证原生渲染风险；P2 解决可靠落盘；P3 最后接入 UI。每一步都有清晰停止点，不需要额外产品选择。

## 风险与缓解

| 风险 | MVP 缓解 |
| --- | --- |
| 图片在每帧解码导致任务栏卡顿 | 仅在配置变化或文件变化时重载，渲染循环只操作内存 RGBA。 |
| 图片写入中断造成半套素材 | 临时文件完成后再替换，并且完整三图成功后才更新配置。 |
| 原图格式/尺寸差异导致 host 支持复杂化 | 前端统一导出固定 64×64 PNG。 |
| 用户外部删除图片 | host 回退默认灯组，设置页显式提示，不自动破坏配置。 |
| Tauri 文件访问权限不足 | P2 前检查现有 capability；只补充该功能必要的最小权限。 |
| 窄窗口裁剪器不可用 | P3 必须以单列布局作为验收项。 |

## 防止范围扩张

- 不新增数据库、云端 API、素材市场或同步服务。
- 不重构 `widget_render` 的状态机、检测器、taskbar attach 路径。
- 不引入通用主题、插件或资源管理框架。
- 不增加图片编辑依赖，除非现有浏览器 Canvas 无法满足固定裁剪需求。
- 不扩展到 GIF、SVG、动画素材或自定义布局。
- 不改动无关设置页面、Win32 fallback UI 或 hooks 生命周期。
- 不把“图片被篡改后自动恢复”做成后台监控服务。

## 延后事项

| 项目 | 延后原因 | 重新评估信号 |
| --- | --- | --- |
| 灯组导入导出 | 不影响本地创建和应用闭环 | 用户明确需要备份或分享。 |
| 预置素材包 | 不影响验证核心创作流程 | 出现重复创作需求。 |
| 状态映射与布局自定义 | 固定绿/黄/红语义足以完成 MVP | 用户反馈固定语义不足。 |
| 多尺寸/高 DPI 策略 | 64×64 源图足以先验证任务栏效果 | 真实截图显示目标 DPI 下清晰度不足。 |
| 素材文件清理 | 当前无需引入误删风险 | 出现明确的存储泄漏证据。 |
