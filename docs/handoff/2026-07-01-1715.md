## 诊断结论更新

- “widget 因为 timer 或 summary 变化本身而自发退出”现在已经被明显降级。
- 先前的高概率怀疑里，至少有两条已被 isolated loop 反证：
  - 无状态变化不会让实例消失
  - synthetic 状态变化也不会让实例消失
- 当前更值得排的方向变成：
  - 真实 live loop 的外部线程事件时序没有落进采样窗口
  - 之前的空白 after 图有一部分可能是 capture / 协调假阴性
  - 真实 `%APPDATA%` 路径下仍可能存在 taskbar host 特有行为，但不再是第一优先级

## 额外修正

- 运行期日志暴露了一个实现问题：
  - `HookSummary.updated_at` 之前会导致每次 `WM_TIMER` 都被判成 summary changed
- 当前已经改成只按显示相关字段决定是否 redraw。
- 重新验证后：
  - baseline 里只会记录 `summary unchanged`
  - fixture replay 里只在 `2 -> 3 -> 4` 时触发 redraw

## 下一步建议

1. 继续用 `real_live` harness，但扩大采样窗口，例如 `0/1000/2000/4000/6000ms`。
2. 让 distinct real thread 事件更早、更确定地落在窗口内，而不是边跑脚本边赌调度。
3. 如果 real live 仍不变，再加一层：
   - 记录 `taskbar_widget_hook.exe list` 的时间点
   - 把外部线程触发和 capture checkpoint 精确对齐
