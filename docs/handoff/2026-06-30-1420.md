# Session Handoff: Stable Win11 Taskbar Custom Content Path

## Session Objectives

本次 handoff 用来交接 `taskbar-widget` 在 Win11 任务栏中稳定显示自定义内容的最终方案。

本轮之前，项目已经证明了三件事：

- 可以稳定找到 `Shell_TrayWnd`、`TrayNotifyWnd` 等任务栏窗口。
- `SetParent`、`MoveWindow`、`WM_PAINT` 都能在 Win32 逻辑层成功。
- 普通 GDI 窗口即使能画、能定位、能出现在 `PrintWindow` 结果中，用户仍然可能在 Win11 真实任务栏上看不到。

本轮最终目标是把问题从“为什么看不见”收敛成“如何稳定显示”。现在的结论是：

> Win11 任务栏自定义内容的稳定路径不是“普通 child HWND + GDI 绘制”，而是“保留 popup 语义 + 在 `SetParent` 前创建 layered surface + 再移动到任务栏区域”。当前稳定组合是 `popup_parented + WS_EX_LAYERED + SetLayeredWindowAttributes(alpha=255, LWA_ALPHA)`。

## Key Adjustment

这次真正让功能稳定可见的关键调整有三步，缺一不可：

1. **不再默认使用纯 `WS_CHILD | WS_VISIBLE` 路径，而是默认使用 `popup_parented`。**

   旧路径会把窗口改成传统 child，然后挂到 `Shell_TrayWnd`。这在 Win32 窗口树和日志里看起来更“正确”，但 Win11 任务栏最终合成层不会稳定呈现它。

   新路径保留窗口创建时的 `WS_POPUP + WS_EX_TOOLWINDOW` 语义，只调用 `SetParent(hwnd, Shell_TrayWnd)`，更接近 TrafficMonitor 的成功模式。

2. **在 `SetParent` 之前给 MVP 窗口加 `WS_EX_LAYERED`。**

   这是最关键的变化。Win11 任务栏不是传统 GDI 父子窗口绘制树，它的最终画面由 DWM / XAML / Composition 参与合成。普通 GDI HWND 可以存在、可以绘制、可以移动到任务栏区域，但不一定进入最终桌面图像。

   加 `WS_EX_LAYERED` 后，窗口获得可被 DWM 合成的独立 surface。用户确认就是这个变量让 `TASKBAR WIDGET` 出现在任务栏右侧。

3. **调用 `SetLayeredWindowAttributes(hwnd, COLORREF(0), 255, LWA_ALPHA)`，先使用完全不透明 alpha。**

   这里故意使用 `alpha=255`，不是为了做透明效果，而是为了只隔离一个变量：窗口是否进入 layered/composition surface。

   这个实验结果说明：问题不在 GDI 绘制内容，也不在坐标，而在“普通 HWND 没有进入 Win11 任务栏最终合成画面”。后续如果要做任务栏背景融合，再从 `LWA_ALPHA=255` 窄化到 `LWA_COLORKEY`，不要一开始就引入透明色和 DComposition。

最终代码路径在 `taskbar-widget/src/taskbar.rs`：

```rust
let ex_style_before = GetWindowLongPtrW(hwnd, GWL_EXSTYLE);
SetWindowLongPtrW(hwnd, GWL_EXSTYLE, ex_style_before | WS_EX_LAYERED);
let layered_ok = SetLayeredWindowAttributes(hwnd, COLORREF(0), 255, LWA_ALPHA).is_ok();
SetParent(hwnd, probe.host_parent);
```

配套修复在 `taskbar-widget/src/main.rs`：

- 注册窗口类时设置 `hCursor = IDC_ARROW`。
- 处理 `WM_SETCURSOR`，避免鼠标移到任务栏模块上显示 loading/busy 光标。

### Stable Runtime Path

当前默认运行路径如下：

```text

  enable_per_monitor_dpi_awareness()
  config = DebugLoopConfig::from_env()
  RegisterClassW(... hCursor = IDC_ARROW ...)
  probe_taskbar(config)
  CreateWindowExW(WS_EX_TOOLWINDOW, WS_POPUP, ...)
  attach_to_taskbar(hwnd, probe, config)
    读取 GWL_EXSTYLE
    设置 WS_EX_LAYERED
    SetLayeredWindowAttributes(alpha=255, LWA_ALPHA)
    SetParent(hwnd, Shell_TrayWnd)
  position_in_taskbar(hwnd, probe, config)

    计算 Shell_TrayWnd client 坐标

  message_loop()
```

## Problems Encountered and Decision Process

### Problem 1: DPI Virtualization Caused Wrong Coordinates

#### Investigation

早期日志显示坐标存在稳定 `0.8` 比例偏差：

```text
move_args=(1634, 0, 160, 48)
module_rect=1307,1104,1435,1142
```

这对应 125% DPI 下进程未声明 DPI awareness，Win32 坐标被虚拟化。

#### Solutions Considered

- 手动乘 DPI 比例。
  - 优点：改动小。
  - 缺点：多显示器和不同 DPI 下容易继续错。
  - 风险：只是补偿症状。

- 在探测和创建窗口前设置 per-monitor DPI awareness。
  - 优点：从根上统一坐标体系。
  - 缺点：需要正确调用 HiDPI API。
  - 风险：必须足够早调用。

#### Final Decision

在启动早期设置 per-monitor DPI awareness。

#### Rationale

修正后任务栏 rect 与模块 rect 对齐到物理像素：

```text
parent_rect=0,1380,2560,1440
module_rect=2084,1380,2244,1440
```

#### Future Implications

后续不要回到固定 96 DPI 或手动比例补偿。

### Problem 2: Logical HWND Success Did Not Equal User Visibility

#### Investigation

旧实现已经能做到：

- 找到 `Shell_TrayWnd`。
- 找到 `TrayNotifyWnd`。
- 调用 `SetParent`。
- 调用 `MoveWindow`。
- 进入 `WM_PAINT` 并绘制文本。
- 在某些 `PrintWindow` 结果里看到内容。

但用户肉眼仍然看不到任务栏内容。

核心事实是：`PrintWindow` 证明窗口能画到内存 DC，不证明 Win11 任务栏最终 DWM/XAML/Composition 图像会包含这个 HWND。

#### Solutions Considered

- 继续把 `PrintWindow` 当 pass/fail。
  - 优点：容易自动化。
  - 缺点：已经和人工观察矛盾。
  - 风险：会产生假阳性。

- 以人工可见作为最终标准，诊断 JSON 只作为辅助证据。
  - 优点：匹配真实目标。
  - 缺点：需要人工参与。
  - 风险：自动化回归较弱。

#### Final Decision

人工可见是最终验收标准。

#### Rationale

真正失败点在最终合成层，而不是窗口树、坐标或 GDI 绘制。

#### Future Implications

后续不要仅凭 `PrintWindow`、`GetParent`、child capture 宣布任务栏可见成功。

### Problem 3: Style and Refresh Matrices Were Not Sufficient

#### Investigation

已经测试过：

```text
TASKBAR_MVP_STYLE_MODE=child
TASKBAR_MVP_STYLE_MODE=popup_parented
TASKBAR_MVP_STYLE_MODE=parent_then_child

TASKBAR_MVP_REFRESH_MODE=none
TASKBAR_MVP_REFRESH_MODE=redraw
TASKBAR_MVP_REFRESH_MODE=topmost_pulse
```

结果：

- `child`：逻辑挂载、定位成功，但人工不可见。
- `popup_parented`：layered 前仍不可见。
- `parent_then_child`：逻辑挂载、定位成功，但人工不可见。
- `redraw` / `topmost_pulse`：不能让内容进入最终任务栏画面。

#### Solutions Considered

- 继续扩大 `SetWindowPos` 和 refresh flag 组合。
  - 优点：实现成本低。
  - 缺点：已有实验信号很弱。
  - 风险：矩阵膨胀但不解释根因。

- 对齐 TrafficMonitor 的 rendering/composition 差异。
  - 优点：TrafficMonitor 是同机成功参考。
  - 缺点：它的差异点很多，需要隔离变量。
  - 风险：容易一次引入太多变量。

#### Final Decision

停止扩大 style/refresh 矩阵，单独验证 layered/composition surface。

#### Rationale

TrafficMonitor 在 Win11 默认配置下通常不会走“纯 GDI、无 layered、无 DComposition”的路径；Rust MVP 失败路径正是这条路径。

#### Future Implications

除非有参考实现证据，不要继续投入 `SWP_*` flag 排列组合。

### Problem 4: Layered Surface Was The Missing Final-Composition Variable

#### Investigation

TrafficMonitor Win11 路径通常会在 `SetParent` 前做透明/layered 处理，或使用 D2D/DComposition。Rust 实验只隔离一个变量：

```text
SetParent 前:
  添加 WS_EX_LAYERED
  SetLayeredWindowAttributes(alpha=255, LWA_ALPHA)
```

`alpha=255` 保持完全不透明，因此这不是透明效果实验，而是“是否进入 layered surface”的实验。

#### Solutions Considered

- `LWA_ALPHA` + `alpha=255`。
  - 优点：只隔离 layered surface；视觉不引入透明变量。
  - 缺点：暂时不融合任务栏背景。
  - 风险：popup 语义需要单独记录。

- 直接 `LWA_COLORKEY`。
  - 优点：更接近 TrafficMonitor 透明色路径。
  - 缺点：同时引入颜色键和背景变量。
  - 风险：成功后难以归因。

- 直接上 D2D/DComposition。
  - 优点：更现代。
  - 缺点：实现复杂度高。
  - 风险：在 layered 已成功时属于过度投入。

#### Final Decision

默认使用 `WS_EX_LAYERED + SetLayeredWindowAttributes(alpha=255, LWA_ALPHA)`。

#### Rationale

用户确认只要启用该变量，任务栏右侧就能看到 MVP。它精确解释了旧矛盾：HWND 能画，但没有可被 Win11 任务栏最终合成层呈现的 surface。

#### Future Implications

当前 MVP 不需要 D2D/DComposition。未来如果要透明融合，应从 `LWA_ALPHA=255` 窄化测试 `LWA_COLORKEY`，不要重开大矩阵。

### Problem 5: Hover Cursor Showed Loading

#### Investigation

第一次 layered 可见后，用户反馈鼠标悬停显示 loading/busy。窗口类此前没有设置 `hCursor`，可能继承了系统当前忙碌光标。

#### Solutions Considered

- 暂不处理。
  - 优点：不影响可见性。
  - 缺点：用户体验明显异常。
  - 风险：看起来像程序卡住。

- 设置 class cursor 并处理 `WM_SETCURSOR`。
  - 优点：标准 Win32 做法。
  - 缺点：多一点消息处理。
  - 风险：很低。

#### Final Decision

设置 `IDC_ARROW`，并在 `WM_SETCURSOR` 中显式 `SetCursor(IDC_ARROW)`。

#### Rationale

这能修复 hover 光标异常，同时不改变 taskbar attach/composition 路径。

#### Future Implications

如果 cursor 问题复现，继续查 hit-test 或 Explorer wait cursor 继承，不要回退 layered 方案。

## Key Decisions and Context

### Stable Implementation Contract

当前稳定显示任务栏自定义内容需要同时满足：

- 启动早期设置 per-monitor DPI awareness。
- 父窗口选择 `Shell_TrayWnd`。
- 位置锚点选择 `TrayNotifyWnd`。
- 创建窗口使用 `WS_POPUP + WS_EX_TOOLWINDOW`。
- 默认 attach 模式使用 `popup_parented`。
- `SetParent` 前设置 `WS_EX_LAYERED`。
- `SetLayeredWindowAttributes(alpha=255, LWA_ALPHA)` 成功。
- 使用 `MoveWindow` 移动到 `Shell_TrayWnd` client 坐标。
- 最终以人工可见为验收标准。

### Why Layered Matters

Win11 任务栏不是传统 GDI 父子窗口绘制树。普通 child HWND 可以存在、移动、绘制，但仍不进入最终桌面图像。layered window 会获得 DWM/taskbar composition 能呈现的 surface，这就是稳定显示的关键。

### Attach Success Semantics

默认 `popup_parented` 下：

```text
success=true
api_ok=true
parent_relation_verified=false
```

含义：

- `SetParent` API 接受了调用。
- `GetParent(hwnd)` 仍返回 `0x0`。
- 对默认稳定路径来说，这不是失败。
- 判断默认路径是否正确，应看 `api_ok=true`、`layered_ok=true`、坐标正确，以及人工可见。

`child` / `parent_then_child` 模式下，`parent_relation_verified=true` 仍然适合用来判断严格 HWND parent relation。

### TrafficMonitor Context

TrafficMonitor 仍是最重要的参考实现：

- Win11 下也使用 `Shell_TrayWnd`。
- 位置也参考 `TrayNotifyWnd`。
- 默认配置通常走 layered 透明色或 DComposition，而不是纯 GDI 普通 child 路径。

本轮 TrafficMonitor 透明色配置对照没有干净完成，因为当前权限无法结束旧进程。后续如果继续做 TrafficMonitor 对照，先手动退出所有 TrafficMonitor 实例，只保留单实例再测。
