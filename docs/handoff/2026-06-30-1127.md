# Win11 Taskbar Widget Session Handoff

## Session Objectives

本轮工作的目标是继续诊断 `taskbar-widget` 在当前 Win11 主力机器上的可见性问题。

核心问题：

- `cargo run` 有完整启动日志。
- `Shell_TrayWnd`、`TrayNotifyWnd`、`Start` 等任务栏窗口句柄都能找到。
- `SetParent`、定位、GDI 绘制从日志和诊断脚本看都成功。
- 用户肉眼仍然看不到任务栏中的 `TASKBAR WIDGET` 模块。

期望结果：

- 在当前 Win11 主显示器任务栏中看到一个固定文本模块 `TASKBAR WIDGET`。
- 模块不是普通桌面浮窗。
- `cargo run` 可重复复现，关闭程序后模块消失。

本轮实际结果：

- 修正了 DPI awareness 问题，坐标从虚拟化坐标切换为真实物理坐标。
- 将诊断证据拆成逻辑挂载、窗口自绘、截图观测、人工可见四层。
- 确认当前最大卡点不是 Rust 绘制或坐标，而是 Win11 任务栏真实合成层没有呈现该 child HWND。
- 尚未实现人工可见。

## Current Implementation Principle

当前实现分为五步：

1. 进程启动时声明 DPI awareness。
2. 创建一个自绘 Win32 窗口，窗口类名为 `TaskbarWidgetWindow`。
3. 探测 Win11 主任务栏窗口树。
4. 将自绘窗口挂到 `Shell_TrayWnd`。
5. 基于 `TrayNotifyWnd` 左侧计算位置，画深色背景和 `TASKBAR WIDGET` 文本。

### Runtime Path Pseudocode

```text
main:
    log "bootstrap start"
    enable_per_monitor_dpi_awareness()

    config = DebugLoopConfig::from_env()
    hinstance = GetModuleHandleW()
    RegisterClassW("TaskbarWidgetWindow", window_proc)

    probe = probe_taskbar(config)
    log_probe(probe)

    hwnd = CreateWindowExW(
        ex_style = WS_EX_TOOLWINDOW,
        class = "TaskbarWidgetWindow",
        style = WS_POPUP,
        x = 0,
        y = 0,
        width = 160,
        height = 48,
        parent = null
    )

    attach = attach_to_taskbar(hwnd, probe)
    layout = position_in_taskbar(hwnd, probe, config)
    ShowWindow(hwnd, SW_SHOW)
    write_diagnostics(...)
    message_loop()
```

### Taskbar Probe Pseudocode

```text
probe_taskbar:
    shell_tray = FindWindowW("Shell_TrayWnd")
    start = FindWindowExW(shell_tray, "Start")
    tray_notify = FindWindowExW(shell_tray, "TrayNotifyWnd")
    rebar = FindWindowExW(shell_tray, "ReBarWindow32")
    task_switch = FindWindowExW(rebar, "MSTaskSwWClass")
    composition_bridge = FindWindowExW(shell_tray, "Windows.UI.Composition.DesktopWindowContentBridge")

    host_parent = shell_tray       # runtime default
    position_anchor = tray_notify  # runtime default

    return TaskbarProbe(...)
```

### Attach Pseudocode

```text
attach_to_taskbar(hwnd, probe):
    previous_parent = GetParent(hwnd)
    style_before = GetWindowLongPtrW(hwnd, GWL_STYLE)

    # Current suspicious implementation detail:
    SetWindowLongPtrW(hwnd, GWL_STYLE, WS_CHILD | WS_VISIBLE)

    set_parent_succeeded = SetParent(hwnd, probe.host_parent)

    style_after = GetWindowLongPtrW(hwnd, GWL_STYLE)
    residual_top_level_style_bits = style_after & TOP_LEVEL_STYLE_MASK
    top_level_style_cleared = residual_top_level_style_bits == 0

    SetWindowPos(hwnd, ..., SWP_FRAMECHANGED | SWP_NOMOVE | SWP_NOSIZE | SWP_NOZORDER | SWP_NOACTIVATE)
    current_parent = GetParent(hwnd)

    return TaskbarAttachResult(...)
```

### Layout Pseudocode

```text
position_in_taskbar(hwnd, probe, config):
    parent_rect = GetWindowRect(Shell_TrayWnd)
    parent_client_rect = GetClientRect(Shell_TrayWnd)
    anchor_rect = GetWindowRect(TrayNotifyWnd)

    module_width = min(160, parent_width)
    margin = 8

    desired_screen_left = anchor_rect.left - module_width - margin
    desired_screen_top = parent_rect.top

    if coord_mode == rect_delta:
        x = desired_screen_left - parent_rect.left
        y = 0
        width = module_width
        height = parent_client_height

    if coord_mode == screen_to_client:
        origin = ScreenToClient(parent, desired_screen_left, desired_screen_top)
        far = ScreenToClient(parent, desired_screen_left + width, desired_screen_top + height)
        x = origin.x
        y = origin.y
        width = far.x - origin.x
        height = far.y - origin.y

    MoveWindow(hwnd, x, y, width, height, repaint = TRUE)
    module_rect = GetWindowRect(hwnd)
```

## Problems Encountered and Decision Process

### Problem 1: 坐标存在 0.8 缩放偏差

#### Investigation

早期日志表现：

- `move_args=(1634, 0, 160, 48)`
- `module_rect=1307,1104,1435,1142`
- `1307 / 1634 ~= 0.8`
- `128 / 160 = 0.8`

该比例对应 125% DPI 下未声明 DPI awareness 的坐标虚拟化。

#### Solutions Considered

Option A: 手动按 DPI 比例乘回物理像素

- Pros: 改动小。
- Cons: 容易在多显示器、不同 DPI 下继续出错。
- Risks: 没有解决根因。

Option B: 启动时声明进程 DPI awareness

- Pros: Win32 坐标体系回到物理像素；与 taskbar rect 更一致。
- Cons: 需要增加 `Win32_UI_HiDpi` feature。
- Risks: 需要确认必须在窗口创建和任务栏探测之前调用。

#### Final Decision

选择 Option B。

#### Rationale

这是坐标偏差的根因修复。修正后日志证明 `move_args` 和 `module_rect` 已对齐。

#### Future Implications

后续如果做多显示器，需要保留 per-monitor DPI 思路，而不是回到固定 96 DPI 假设。

### Problem 2: `PrintWindow` 截图和人工可见性矛盾

#### Investigation

诊断脚本曾输出 child capture 里有 `TASKBAR WIDGET`，并且 `PrintWindow(Shell_TrayWnd)` 的图里也可能出现 `TASKBAR WIDGET`。但用户肉眼看不到任务栏模块。

关键解释：

- `PrintWindow` 不是实际屏幕截图。
- `PrintWindow` 会请求目标 HWND 自己绘制到内存 DC。
- Win11 任务栏真实显示大量依赖 XAML / DWM / Composition，不一定走传统 GDI/WM_PRINT 路径。
- 因此 `PrintWindow` 图里看见 `TASKBAR WIDGET` 只证明 child HWND 能画，不证明它被真实任务栏合成层显示。

用户提出的问题：

- “为什么截图像任务栏，却没有右侧时间？”

解释：

- 那张图是 `Shell_TrayWnd` 的 `PrintWindow` 捕获，不是桌面像素截图。
- Win11 的时间、托盘等区域很多不是通过 `Shell_TrayWnd` 的 GDI 绘制输出。
- 所以 `PrintWindow` 图可能缺少真实任务栏右侧时间等元素。

#### Solutions Considered

Option A: 继续以 `PrintWindow` 为 pass/fail

- Pros: 自动化简单。
- Cons: 已经产生误导。
- Risks: 会把“自绘成功”误判为“用户可见”。

Option B: 同时捕获真实屏幕像素

- Pros: 能更接近人工观察。
- Cons: 在 Codex/远程桌面/焦点环境下仍可能不可靠。
- Risks: 不能完全取代用户肉眼验证。

Option C: 把人工观察设为主真相源

- Pros: 直接对应 MVP 目标。
- Cons: 需要用户参与验证。
- Risks: 自动回归能力弱。

#### Final Decision

选择 B + C：脚本保留辅助事实，人工观察作为最终判断。

#### Rationale

当前问题是 Win11 合成层是否真实可见，而不是子窗口是否能画。自动截图只能提供辅助证据。

#### Future Implications

后续文档和 checklist 不应把 `VisualPass` 等同于产品成功。应明确区分：

- `AttachSuccess`: 逻辑父子关系成立。
- `LayoutMoved`: 坐标移动成功。
- `RenderPass`: child HWND 自绘成功。
- `VisualPass`: 捕获工具观测到变化。
- `ManualVisible`: 用户肉眼可见。

### Problem 3: `SetParent` 成功但人工不可见

#### Investigation

最新用户日志显示：

- `SetParent` 成功。
- `current_parent=Shell_TrayWnd`。
- `style_after=0x54000000`。
- `top_level_cleared=true`。
- `module_rect=2084,1380,2244,1440` 位于任务栏区域。
- 用户仍看不到模块。

这说明问题不在窗口句柄不存在，也不在坐标越界。

最大可疑点：

- 当前代码在 `SetParent` 前强行执行：

```rust
SetWindowLongPtrW(hwnd, GWL_STYLE, (WS_CHILD.0 | WS_VISIBLE.0) as isize);
```

这会清掉 `WS_POPUP` 等顶层窗口语义。

TrafficMonitor 参考路径中，MFC dialog 调用 `ModifyStyleEx(0, WS_EX_TOOLWINDOW)`，然后 `SetParent(this->m_hWnd, GetParentHwnd())`。从已查到代码看，没有发现它在 `SetParent` 前显式把主 style 改成纯 `WS_CHILD | WS_VISIBLE`。

#### Solutions Considered

Option A: 保持纯 `WS_CHILD | WS_VISIBLE`

- Pros: 符合传统 SetParent 推荐语义。
- Cons: 当前人工不可见。
- Risks: Win11 taskbar composition 可能不接受这种 child 呈现方式。

Option B: 对齐 TrafficMonitor，保留 `WS_POPUP` style，只调用 `SetParent`

- Pros: 更接近已知产品实现。
- Cons: 可能留下 top-level style residual。
- Risks: 需要验证是否会出现普通桌面浮窗或 Alt-Tab/任务栏图标副作用。

Option C: 挂到 `composition_bridge` 或 `input_site`

- Pros: 更接近 Win11 真实合成层。
- Cons: 之前矩阵跑过多个父窗口候选，没有得到稳定人工成功。
- Risks: 容易扩大搜索空间，偏离 MVP。

Option D: 直接改用 DComposition / XAML island 思路

- Pros: 理论上更贴近 Win11 任务栏合成技术。
- Cons: 超出 MVP，复杂度高。
- Risks: 会把嵌入问题和渲染平台问题混在一起。

#### Final Decision

尚未实施最终修复。推荐下一步优先测试 Option B。

#### Rationale

当前 evidence chain 已经排除 DPI 和基本坐标问题。与参考实现相比，最明显、最小、可验证的差异就是 style 改写策略。

#### Future Implications

如果保留 `WS_POPUP` 后人工可见，需要重新定义诊断字段：`top_level_cleared=false` 不一定是失败。成功标准应从“清掉所有顶层 style”改成“没有普通浮窗副作用且任务栏内可见”。

## Key Decisions and Context

### Win11 主路径

当前主路径固定为：

- 父窗口：`Shell_TrayWnd`
- 定位锚点：`TrayNotifyWnd`
- 坐标模式：`rect_delta`
- 模块宽度：`160`
- 模块高度：父窗口 client height，当前为 `60`
- 绘制：GDI 深色背景 + 白色 `TASKBAR WIDGET`

### 为什么不继续扩大父窗口矩阵

此前已经尝试过 `rebar`、`task_switch`、`composition_bridge` 等方向。当前目标是 MVP 可见闭环，继续扩大矩阵会增加变量数量，不会直接解释“Shell parent + 正确坐标 + 自绘成功但人工不可见”的矛盾。

### 为什么必须保留人工验证

Win11 任务栏不是传统 GDI 子窗口树直接等于屏幕结果。`PrintWindow` 和真实桌面合成不是同一条路径。当前 MVP 成功标准是用户肉眼能看到，所以人工验证是最终真相源。

### 当前最重要的工程判断

当前实现已经证明：

- Rust 调 Win32 可行。
- 任务栏窗口树可探测。
- DPI 问题已解决。
- 自绘窗口能绘制。
- `SetParent` 能成功。
- 坐标能落在任务栏区域。

当前尚未证明：

- Win11 任务栏真实合成层会显示该 child HWND。

## Discussion Summary for Humans

如果要和别人讨论，可以直接聚焦这句话：

当前不是 Rust 绘制失败，也不是找不到任务栏，也不是 DPI 坐标错误。现在的矛盾是：Win32 逻辑层显示窗口已经是 `Shell_TrayWnd` 的 child，且 child 自己能画；但 Win11 真实任务栏由 Composition/XAML/DWM 层合成，未必会把这个 child HWND 呈现到用户眼前。`PrintWindow` 能看到 `TASKBAR WIDGET` 是因为它请求窗口自己画到内存 DC，不等价于桌面最终画面。

当前最小可验证假设：

```text
TrafficMonitor 成功路径可能并没有把窗口主 style 强制改成纯 WS_CHILD。
我们的代码清掉 WS_POPUP 后，虽然 SetParent 成功，但可能失去了 Win11 taskbar 合成层愿意显示它的窗口语义。
```

下一步建议只测这个变量，不引入 D2D、透明、多父窗口矩阵或完整重构。
