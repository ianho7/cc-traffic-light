## Problems Encountered and Decision Process

### Problem

之前对 Codex 的判断基于 `notify`，导致结论偏差，看起来像“Codex hooks 拿不到状态”。

### Investigation

- 先做了 `notify` wrapper 探针。
- 真实结果显示 `notify` 的 stdin 为空，没有 `session_id/thread_id/turn_id`。
- 随后查阅官方 Codex hooks 文档，并用项目级 `.codex/hooks.json` + dump 脚本验证正式 lifecycle hooks。
- 真实 lifecycle dump 证据显示：`PreToolUse`、`PostToolUse`、`Stop` 都有结构化 JSON，且包含 `session_id` 和 `turn_id`。

### Solutions Considered

- Description: 继续依赖 `notify`
  - Pros: 不用重新配 lifecycle hooks
  - Cons: 没有任务 identity
  - Risks: 多任务状态污染

- Description: 直接把 lifecycle hooks 指向真实状态写入
  - Pros: 直达目标链路
  - Cons: 若 payload 结构未知，容易误写状态
  - Risks: 需要重新 trust，且一旦规则不稳会污染状态文件

- Description: 先用 lifecycle dump 采样，再切到真实状态写入
  - Pros: 风险最低，证据充分
  - Cons: 多一步切换
  - Risks: 需要维护额外测试配置

### Final Decision

先用项目级 lifecycle dump 验证真实 payload，再把 Codex 正式接入主状态路径。

### Rationale

这条路径兼顾了正确性和最小风险。它避免了用低保真 `notify` 做过度推断，也避免了在没有证据的前提下直接写入真实状态。

### Future Implications

- 现在已经有足够证据执行 P0。
- 下一步切换 `.codex/hooks.json` 时，Codex 会因为 hook 定义变化要求重新 trust。
- 最终产品不应该依赖项目级 hooks，而应迁移到用户级全局 hooks。

### Problem

现有 `hook_rules.rs` 对正式 Codex 事件的覆盖不完整，新加的 `SessionStart` 如果按“未知事件”处理会错误落到 `working`。

### Investigation

- 读取现有 `hook_rules.rs`，确认未知事件默认映射为 `working`。
- 结合官方事件列表和真实 dump，确认至少应对 `SessionStart`、`PostToolUse`、`SubagentStart`、`PreCompact`、`PostCompact` 做保守处理。

### Solutions Considered

- Description: 保持未知事件默认 `working`
  - Pros: 改动小
  - Cons: `SessionStart` 会污染状态
  - Risks: UI 会无端亮起工作态

- Description: 先补基础保守映射
  - Pros: 风险低，足以支撑下一轮真实接入
  - Cons: 后续仍需根据真实 payload 微调
  - Risks: `PostToolUse` 失败仍可能过于保守

### Final Decision

补充 `SessionStart -> idle`，并将 `PostToolUse`、`PreCompact`、`PostCompact`、`SubagentStart` 保守映射到 `working`。

### Rationale

这样可以保证示例配置与真实 Codex 事件集一致，同时避免最明显的状态误判。

### Future Implications

后续如果拿到 `PostToolUse` 失败的真实字段证据，仍需要补充更精确的错误映射。
