## Problems Encountered and Decision Process

## Problem 1: 运行日志正常，但任务栏肉眼不可见

### Investigation

用户多次运行 `cargo run`，日志稳定显示：

- `Shell_TrayWnd` 能找到。
- `TrayNotifyWnd` 能找到。
- `Start` 能找到。
- `SetParent` 成功。
- `MoveWindow` 成功。
- `module_rect` 落在任务栏范围内。

典型日志：

```text
shell_tray: hwnd=0x100FC rect=0,1380,2560,1440
tray_notify: hwnd=0x10104 rect=2252,1380,2560,1440
attach result success=true current_parent=0x100FC
layout result moved=true module_rect=2084,1380,2244,1440
```

但用户确认：任务栏没有出现 `TASKBAR WIDGET`，也没有闪现。

### Solutions Considered

- 坐标错误：已通过 `module_rect` 和任务栏范围验证，基本排除。
- DPI 错误：已设置 per-monitor v2，日志显示 `dpi=120 scale=1.25`，坐标也匹配 2560x1440/125% 场景，基本排除。
- 没有绘制：`PrintWindow` / child capture 曾能证明 HWND 内部有绘制，但这不能证明最终桌面合成可见。
- 父窗口错误：`Shell_TrayWnd` 与 TrafficMonitor Win11 路径一致，暂不作为主嫌疑。

### Final Decision

将问题拆成四层：

1. 逻辑挂载：`SetParent` / `GetParent`。
2. 坐标布局：`MoveWindow` / `module_rect`。
3. HWND 自绘：`WM_PAINT` / child capture。
4. 最终桌面合成：用户肉眼可见。

当前失败点在第 4 层。

### Future Implications

不能再把 `PrintWindow` 成功当成最终可见成功。Win11 任务栏包含 XAML/DWM/Composition 层，普通 child HWND 可能存在于窗口树中但不被最终合成画面采用。

## Problem 2: `WS_CHILD | WS_VISIBLE` 是否导致不可见

### Investigation

最初最大嫌疑是 Rust 代码在 `SetParent` 前强制执行：

```rust
SetWindowLongPtrW(hwnd, GWL_STYLE, WS_CHILD | WS_VISIBLE);
```

而 TrafficMonitor 的成功路径看起来是 MFC dialog 初始化后 `ModifyStyleEx(WS_EX_TOOLWINDOW)`，再 `SetParent(this->m_hWnd, GetParentHwnd())`，没有明显在 `SetParent` 前强制改为纯 `WS_CHILD | WS_VISIBLE`。

因此加入 `TASKBAR_MVP_STYLE_MODE`：

- `child`：当前基线，`SetParent` 前改成 `WS_CHILD | WS_VISIBLE`。
- `popup_parented`：保留 `WS_POPUP`，只 `SetParent`。
- `parent_then_child`：先 `SetParent`，再改成 `WS_CHILD | WS_VISIBLE`，模拟“先挂载后变 child”的路径。

### Results

#### `child`

用户运行：

```powershell
$env:TASKBAR_MVP_STYLE_MODE="child"
cargo run
```

结果：

```text

attach success=true
current_parent=0x100FC
style_before=0x84000000
style_after=0x54000000
top_level_cleared=true
module_rect=2084,1380,2244,1440
```

人工结果：不可见。

#### `popup_parented`

用户运行：

```powershell
$env:TASKBAR_MVP_STYLE_MODE="popup_parented"
cargo run
```

早期日志显示：

```text
api returned ok-like result, but current_parent=0x0
style_after=0x84000000
top_level_cleared=false
```

后续代码已修正判定逻辑：`SetParent` API 返回不等于实际 attach 成功，必须看 `GetParent`。

人工结果：不可见。

#### `parent_then_child`

用户运行：

```powershell

```

结果：

```text

style_before=0x84000000
```

人工结果：不可见。

### Final Decision

“`SetParent` 前改 style 导致不可见”不是根因。`parent_then_child` 已经验证 style 顺序变化后仍不可见。

### Future Implications

不要继续只围绕 `WS_CHILD` / `WS_POPUP` 顺序做小修小补。该方向已被当前机器上的人工观察证伪。

## Problem 3: 是否需要显式重绘或 z-order pulse

### Investigation

第二轮假设是：窗口已经是 child，但被 Win11 任务栏 composition 层盖住，或者需要显式重绘/置顶才能进入最终画面。

加入 `TASKBAR_MVP_REFRESH_MODE`：

- `none`：不做额外刷新。
- `redraw`：三次执行 `MoveWindow + InvalidateRect + UpdateWindow`，间隔 500ms。
- `topmost_pulse`：三次执行 `SetWindowPos(HWND_TOP, x, y, w, h, SWP_NOACTIVATE | SWP_SHOWWINDOW)`，再 `InvalidateRect + UpdateWindow`，间隔 500ms。

注意：`topmost_pulse` 明确不带 `SWP_NOZORDER`，目的是测试 z-order 影响。

### Results

#### `redraw`

用户运行：

```powershell

$env:TASKBAR_MVP_REFRESH_MODE="redraw"
cargo run
```

日志：

```text

refresh redraw attempt=2 moved=true invalidated=true updated=true
refresh redraw attempt=3 moved=true invalidated=true updated=true
```

人工结果：不可见。

#### `topmost_pulse`

用户运行：

```powershell
$env:TASKBAR_MVP_STYLE_MODE="parent_then_child"
$env:TASKBAR_MVP_REFRESH_MODE="topmost_pulse"
```

日志：

```text
refresh topmost_pulse attempt=2 positioned=true invalidated=true updated=true

```

人工结果：不可见。

### Final Decision

显式重绘和普通 z-order pulse 不是根因。当前 HWND 已经能 move、invalidate、update、top-order pulse，但仍没有进入 Win11 任务栏最终合成画面。

### Future Implications

下一步不要继续扩大 `SetWindowPos` flag 组合矩阵。继续调 `SWP_*` 的边际价值很低，除非有 TrafficMonitor 对照证明它依赖某个具体 flag。

## Problem 4: TrafficMonitor 成功路径与 Rust MVP 差异

### Investigation

参考了 `D:/project/TrafficMonitor/TrafficMonitor/TaskBarDlg.cpp` 和 `Win11TaskbarDlg.cpp`。

TrafficMonitor 关键路径：

```cpp
CDialogEx::OnInitDialog();
SetWindowText(TASKBAR_WINDOW_NAME);
ModifyStyleEx(0, WS_EX_TOOLWINDOW);
m_hTaskbar = FindTaskbarHandle(...);
ApplyWindowTransparentColor();
InitTaskbarWnd();
SetParent(this->m_hWnd, GetParentHwnd());
SetDPI(...);
SetTextFont();
CalculateWindowSize();
AdjustWindowPos(true);
SetBackgroundColor(...);
SetTimer(TASKBAR_TIMER, 1000, NULL);
```

Win11 parent：

```cpp
HWND CWin11TaskbarDlg::GetParentHwnd()
{
    return m_hTaskbar;
}
```

Win11 position 逻辑：

- `m_hNotify = FindWindowEx(m_hTaskbar, 0, L"TrayNotifyWnd", NULL)`。
- 以通知区左侧为参考，把窗口移动到通知区左边。

### Observed Differences

Rust MVP 当前路径：

```rust
CreateWindowExW(WS_EX_TOOLWINDOW, WS_POPUP, ...)
attach_to_taskbar()
position_in_taskbar()
ShowWindow()
refresh_visibility()
message_loop()
```

TrafficMonitor 路径差异：

- 使用 MFC `CDialogEx`，不是裸 `CreateWindowExW` 普通窗口。
- dialog 已完整 `OnInitDialog` 后再 `SetParent`。
- 有透明色处理和背景色设置。
- 有字体、DPI、窗口尺寸计算后再 `AdjustWindowPos(true)`。
- 有 timer 周期刷新。
- 可能使用 D2D / DComposition / GDI double buffer 绘制路径。

### Final Decision

TrafficMonitor 的父窗口和锚点思路与 Rust 基本一致，因此“找错父窗口”暂不是主要突破口。更可能的差异是窗口创建语义、dialog 生命周期、透明/绘制栈、或当前 Win11 构建是否仍允许这种插入方式。

## Key Decisions and Context

### Manual Visibility Is The Source Of Truth

当前项目必须区分：

- `SetParent` 成功：窗口树逻辑成立。
- `module_rect` 正确：坐标成立。
- child capture 有内容：窗口能画。
- 用户肉眼可见：最终桌面合成成立。

本问题只在最后一层失败。

### Do Not Trust `PrintWindow` As Final Evidence

`PrintWindow(Shell_TrayWnd)` 或 child capture 能看到内容，只能说明 HWND 自己能画到内存 DC。Win11 任务栏最终画面由 DWM/XAML/Composition 参与，内存 DC 绘制结果不等价于桌面最终像素。

### TrafficMonitor Is The Best Differential Oracle

同机 TrafficMonitor 对照是下一步最高价值实验，因为它能把问题分成两类：

- 系统/构建不支持：TrafficMonitor 也不可见。
- Rust 语义差异：TrafficMonitor 可见而 Rust 不可见。

没有这个对照，继续扩 Rust 参数矩阵容易陷入低价值猜测。
