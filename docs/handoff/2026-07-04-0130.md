## Executive Summary

- 当前仓库正在把 settings UI 从 `Slint` 迁移到 `Tauri + React + PNPM`，但仍保留 `taskbar-widget` 作为 Win32 宿主、tray、widget 和 detector 主循环。
- 迁移的主要代码路径已经接上：tray 默认先走 Tauri settings，宿主与 settings 之间通过 named pipe IPC 通信，共享配置和状态逻辑已经下沉到 `shared-core`。
- 本文前半部分记录的是最初阻塞验收的启动错误：宿主可执行文件 [taskbar-widget.exe](/D:/project/cc-traffic-light/target/debug/taskbar-widget.exe) 在启动最早期报错：

```text

无法定位程序输入点 TaskDialogIndirect 于动态链接库
D:\project\cc-traffic-light\target\debug\taskbar-widget.exe 上。
```

- 这类错误发生在 PE 装载器解析导入表的阶段，晚于文件被创建、早于 Rust `main()` 执行。因此宿主进程相关的运行期验证无法继续做实。
- 当时已坐实两件关键事实：
  - 这份 exe 的 import table 确实直接导入了 `comctl32.dll!TaskDialogIndirect`
  - 这份 exe 也确实带有正确的 `RT_MANIFEST / ID=1` manifest，内容包含 `Microsoft.Windows.Common-Controls` v6 依赖
- 基于后续诊断和验证，当前更准确的结论已经更新为：
  - 根因不是 manifest，而是 `cargo build --workspace` 时，Tauri 侧依赖链把 `muda/common-controls-v6` 特性统一进了宿主产物
  - 该特性会把 `TaskDialogIndirect` 静态导入进宿主 exe，导致 loader 阶段失败
  - 只要改为分别构建 `taskbar-settings-tauri` 和 `taskbar-widget`，并以单包构建后的宿主产物做验证，`TaskDialogIndirect` 就会从宿主导入表消失，生命周期验证可通过

## The Blocking Problem

### Observed Symptom

用户在运行宿主时看到弹窗：

```text
taskbar-widget.exe - 无法找到入口
无法定位程序输入点 TaskDialogIndirect 于动态链接库
D:\project\cc-traffic-light\target\debug\taskbar-widget.exe 上。
```

### Why It Matters

这个错误发生在宿主启动阶段，因此以下运行期验证无法可靠完成：

- 从 tray 默认打开 Tauri settings
- 多次打开/关闭 settings 是否复用单进程
- 宿主退出时是否回收 settings 子进程
- Tauri settings 异常退出后 tray 再次打开是否恢复
- 真实运行期是否存在日志刷屏、轮询抖动或窗口卡顿

### Why It Is a Loader-Stage Failure

- 宿主入口 `main()` 的第一阶段就会执行 `init_runtime_log()`
- 但当前出错时没有生成 runtime log
- 因此失败发生在 `main()` 之前，最符合 PE 装载器解析 import table 的阶段

## Follow-Up Resolution

这一节记录后续线程中已经验证过的修复与结论，优先级高于本文前面“剩余猜测”部分。

### Root Cause

当前已验证的根因是：

- 问题不是 manifest
- 问题是 `cargo build --workspace` 时，workspace 里的 Tauri 成员把 `muda/common-controls-v6` 合并进了宿主侧最终链接产物
- 结果就是：
  - 单独构建 `taskbar-widget` 时，宿主导入表里没有 `TaskDialogIndirect`
  - 工作区整体构建后，宿主导入表里会重新出现 `TaskDialogIndirect`

### What Was Changed

后续线程中，当前工作树已经落地了以下修复：

1. [validate-tauri-settings-lifecycle.ps1](/D:/project/cc-traffic-light/taskbar-widget/scripts/validate-tauri-settings-lifecycle.ps1)
   - 不再执行 `cargo build --workspace --offline`
   - 改为分别构建：
     - `cargo build -p taskbar-settings-tauri`
     - `cargo build -p taskbar-widget`
   - 并保证 host 最后单独构建，避免 feature 污染
   - 同时补强了窗口定位逻辑：
     - 标题优先
     - 失败则按 settings 进程 PID 找顶层窗口
   - 结束验证时不再强杀宿主，而是给宿主发 `WM_CLOSE`，让 `WM_DESTROY` 路径回收 settings 子进程

2. [main.rs](/D:/project/cc-traffic-light/taskbar-widget/src/main.rs)
   - `tray_icon::add_tray_icon(...)` 失败时不再直接返回错误退出
   - 改为记录 runtime log 后继续运行
   - 这样无 Explorer/taskbar 的验证环境不会把 tray 初始化失败误判成宿主整体启动失败

3. [task-TSM-VAL-04-20260704-114022.md](/D:/project/cc-traffic-light/docs/reflections/task-TSM-VAL-04-20260704-114022.md)
   - 记录了本次 loader 阻塞的最终诊断结论和验证流程

### Verified Outcome

后续线程里，以及当前工作树复核时，已经验证到：

- `cargo check -p taskbar-widget --offline` 通过

## Hypotheses That Have Been Ruled Out

以下方向目前基本可以排除：

- 不是因为 `app.manifest` 文件里把 `processorArchitecture` 写成了 `x86`
- 不是因为 manifest 只是字符串，没有被登记进 PE 资源表

- 不是因为 `build.rs` 用了 GNU 不支持的链接参数语法，当前工具链是 `x86_64-pc-windows-msvc`
- 不是因为“当前机器 Common Controls v6 manifest 机制本身必然失效”，最小 probe 已经反向证明该机制可工作
