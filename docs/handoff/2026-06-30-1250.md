# Session Handoff: Win11 Taskbar Widget Layered Visibility Result

## Session Objectives

本轮目标是验证并修复 `taskbar-widget` 在 Win11 任务栏最终合成层不可见的问题。

进入本轮前已经确认：

- `SetParent` / `MoveWindow` / `WM_PAINT` 在逻辑层面都能工作。
- `style_mode=child|popup_parented|parent_then_child` 和 `refresh_mode=none|redraw|topmost_pulse` 已排除为单独根因。
- 同机 TrafficMonitor 默认配置能显示在任务栏，说明系统构建本身不阻止此类任务栏窗口。

本轮核心问题：

> Rust MVP 的纯 GDI 普通窗口虽然能自绘并落在任务栏区域，但没有进入 Win11 任务栏最终桌面合成画面。TrafficMonitor 默认 Win11 路径几乎总会使用 layered 透明色或 DComposition，因此需要单独验证 layered surface 是否是缺失变量。

最终结论：

> `WS_EX_LAYERED + SetLayeredWindowAttributes(alpha=255, LWA_ALPHA)` 是本轮可见性突破口。`TASKBAR_MVP_LAYERED=opaque` 与 `TASKBAR_MVP_STYLE_MODE=popup_parented` 组合下，用户肉眼确认任务栏右侧出现了 MVP 窗口。layered 变量是当前“临门一脚”。

### Rust layered 受控实验

修改文件：

- `taskbar-widget/src/taskbar.rs`
- `taskbar-widget/src/main.rs`

新增环境变量：

```powershell
TASKBAR_MVP_LAYERED=off|opaque
```

实现位置：

- 在 `attach_to_taskbar()` 中、`SetParent` 调用之前执行 layered 设置。
- `opaque` 模式只加 `WS_EX_LAYERED` 并设置 `LWA_ALPHA` + alpha `255`，不启用 color key。

关键实现行为：

```rust
let ex_style_before = GetWindowLongPtrW(hwnd, GWL_EXSTYLE);
SetWindowLongPtrW(hwnd, GWL_EXSTYLE, ex_style_before | WS_EX_LAYERED);
let layered_ok = SetLayeredWindowAttributes(hwnd, COLORREF(0), 255, LWA_ALPHA).is_ok();
let ex_style_after = GetWindowLongPtrW(hwnd, GWL_EXSTYLE);
```

日志和诊断 JSON 新增字段：

- `layered`
- `ex_style_before`
- `ex_style_after`
- `layered_attempted`
- `layered_ok`
- `layered_last_error`
- `parent_relation_verified`

实验命令：

```powershell
$env:TASKBAR_MVP_STYLE_MODE="popup_parented"
$env:TASKBAR_MVP_LAYERED="opaque"
cargo run
```

### Default Path Convergence

基于可见结果，默认路径已收敛为刚验证成功的组合：

- `TASKBAR_MVP_STYLE_MODE` 默认：`popup_parented`
- `TASKBAR_MVP_LAYERED` 默认：`opaque`

仍保留回归对照：

```powershell
$env:TASKBAR_MVP_STYLE_MODE="child"
$env:TASKBAR_MVP_LAYERED="off"
```

用于复现旧的普通 GDI / child 路径。

### Cursor Fix

针对 hover loading 状态，`main.rs` 做了两处修复：

- 注册窗口类时设置 `hCursor = IDC_ARROW`。
- 显式处理 `WM_SETCURSOR`，调用 `SetCursor(IDC_ARROW)` 并返回 handled。

目的：

- layered / popup 语义下不再继承或保留系统 busy cursor。
- 鼠标移到 MVP 窗口上应显示普通箭头。

## Problems Encountered and Decision Process

### Problem 1: TrafficMonitor 透明色开关对照无法形成干净实验

#### Investigation

找到了正确配置文件，也成功把：

```ini
taksbar_transparent_color_enable = true
```

临时改成：

```ini
taksbar_transparent_color_enable = 0
```

但旧 TrafficMonitor 进程无法被当前权限结束。启动新实例后出现多个同名进程，无法判断哪个实例读取了哪个配置，也无法确认任务栏窗口来自哪个实例。

#### Final Decision

不把 Step 1 作为本轮有效证据。配置已经恢复，避免污染用户环境。

#### Future Implications

如果仍要完成 TrafficMonitor 交叉验证，需要用户手动从托盘退出 TrafficMonitor，或在管理员权限/同一桌面会话中重跑：

1. 退出所有 TrafficMonitor 实例。
2. 改 `taksbar_transparent_color_enable = 0`。
3. 启动单实例。
4. 人工观察是否仍可见。
5. 恢复配置。

### Problem 2: Rust 普通 child GDI 路径不进最终合成

#### Investigation

之前的事实链已经证明：

- HWND 自己能绘制。
- 坐标落在任务栏区域。
- `PrintWindow` 不能代表最终桌面合成。

本轮新增的唯一变量是 layered surface。`SetLayeredWindowAttributes(alpha=255)` 不改变透明度，只隔离“是否进入 layered / 独立合成 surface”。

#### Result

一旦加 `WS_EX_LAYERED` 并设置 alpha 255，用户肉眼确认任务栏右侧出现 MVP。

#### Final Decision

把 `LayeredMode::Opaque` 收敛为默认路径。

#### Future Implications

当前不需要直接跳 D2D / DirectComposition。`WS_EX_LAYERED + LWA_ALPHA=255` 已经足以让最小 MVP 进入 Win11 最终合成画面。

### Problem 3: Hover 显示 loading 光标

#### Investigation

首次 layered 可见后，用户反馈鼠标移上去是 loading 状态。当前窗口类此前没有设置 `hCursor`，裸 Win32 窗口可能继承或保留当前光标状态。

#### Final Decision

设置窗口类 cursor，并处理 `WM_SETCURSOR` 返回标准箭头。

#### Future Implications

如果用户最终确认 hover 仍异常，再检查是否是窗口线程忙、Explorer wait cursor 继承，或 layered popup 与任务栏 hover hit-test 的交互问题。但第一修复应是标准 cursor 设置。
