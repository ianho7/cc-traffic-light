## Problems Encountered and Decision Process

### Problem

旧版 widget 只会显示整块背景和长文本，任务栏尺寸下状态识别效率低。

### Investigation

- 阅读 `main.rs` 与 `agent_state.rs`
- 确认 widget 只消费 `global_summary`
- 对照 P3 README 范围边界

### Solutions Considered

- 保持纯文本
  - Pros: 改动最小
  - Cons: 视觉信号弱
  - Risks: P3 目标无法成立
- 纯圆点
  - Pros: 最简
  - Cons: 语义和计数信息损失太大
  - Risks: 后续又要补更多 UI 解释层
- 圆点 + 紧凑标签
  - Pros: 形状与语义兼顾
  - Cons: 仍需控制长度
  - Risks: 布局过满时需要截断

### Final Decision

采用“左侧状态灯 + 右侧紧凑标签 + 右上角 stale 告警点”。

### Rationale

这是在不扩大 widget、不改宿主路径的前提下，信息密度和可见性最平衡的方案。

### Future Implications

- 如果后续进入多任务或按 agent 视图，这套布局需要重新设计
- 当前布局只适合 P3 的全局 summary MVP

### Problem

真实 Codex hook 中，失败请求落到 `done`，批准后的请求停在 `waiting`。

### Investigation

- 根据真实对话观察到：
  - 成功：`run -> done`
  - 失败：`run -> done`
  - 批准路径：`run -> wait -> wait`
- 阅读 [hook_rules.rs](/D:/project/cc-traffic-light/taskbar-widget/src/hook_rules.rs)
- 发现：
  - `Stop` 只要 previous 是 `waiting` 或文本像等待，就继续判成 `waiting`
  - 失败态只有 `StopFailure` 才进 `error`
  - 项目 `.codex/hooks.json` 当前并未挂单独 Codex 失败态事件

### Solutions Considered

- 继续做更多 UI 测试
  - Pros: 无需改代码
  - Cons: 无法解决真实语义错误
  - Risks: 在错误状态映射上浪费时间
- 修改 `taskbar.rs` / 轮询机制
  - Pros: 可以尝试改变表现
  - Cons: 问题不在这里
  - Risks: 破坏稳定宿主路径
- 收紧 `hook_rules.rs`
  - Pros: 直接命中根因
  - Cons: `Stop -> error` 仍带启发式
  - Risks: 需要后续真实 payload 继续校准

### Final Decision

直接修改 `hook_rules.rs`：

- 移除 `Stop` 的 waiting 保留逻辑
- 给 `Stop` 增加失败文本识别
- 补 `PostToolUseFailure` / `ToolUseFailure` 到 `error`

### Rationale

问题表现和代码路径高度一致，规则层是最小修复面。

### Future Implications

- 当前修复已满足 P3 需要
- 若后续拿到更明确的 Codex 失败事件，应把文本启发式替换成显式事件映射
