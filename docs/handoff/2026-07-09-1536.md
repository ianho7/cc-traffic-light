## 会话目标

原始问题是：CC Traffic Light 的 `taskbar_widget_hook.exe` 无法从 Claude Code 接收到 hook 事件（Codex 正常工作）。

此前已有 `docs/handoff/claude-code-hooks-investigation.md` 记录了初步调查，结论是"Claude Code command hooks 无法写入 state.json"，建议修改 understand-anything plugin 的 hooks.json 作为 workaround。

**本次会话的目标：**
1. 验证或反驳之前调查的结论（settings.json/settings.local.json hooks 启动时不加载）
2. 测试替代假设（workspace trust、JSON/BOM 问题、路径问题）
3. 找到 Claude Code 可用的 hooks 配置方案
4. 部署工作配置

## 已完成工作

### 1. 第一轮调查——方法有缺陷

使用 `claude --debug-file <path>` 捕获 hook 执行日志。多轮测试发现：
- `Registered 2 hooks from 1 plugins`（understand-anything plugin 的 hooks，确认这是 plugin 注册机制）
- `Hooks: Found 0 total hooks in registry`（**误判**为"settings hooks 未加载"）
- File watcher 确认同时监控 settings.json 和 settings.local.json

**关键错误：** 使用 `cmd.exe /c "echo fired > \"%TEMP%\\cc-hook-test.txt\""` 作为 marker——`%TEMP%` 环境变量在 exec form 的 spawn 上下文中不可用，导致 marker 文件从未被创建，误判为 hooks 没触发。

**错误结论发布到 handoff 文档：**"settings.json hooks 启动时不加载。"

### 2. 二进制代码分析（已撤回）

用 `Select-String` 从 246MB 的 `claude.exe` Bun bundle 中提取函数名。
- 找到 `Jwy`（file watcher）、`Cst`（ConfigChange handler）、`Kkd`（plugin hook loader）、`shouldSkipHookDueToTrust`
- 构建了"启动路径 vs watcher 路径"的双路径调用图理论
- **此分析不可靠**——minified Bun bundle 的函数名映射不能用于构建准确的调用图

### 3. 替代假设的排除

| 假设 | 结果 | 方法 |
|---|---|---|
| JSON/BOM 编码问题 | **已排除** | `[System.IO.File]::ReadAllBytes` 确认所有 settings 文件首字节为 `0x7B`（`{`），无 UTF-8 BOM。`ConvertFrom-Json` 验证通过 |
| Workspace trust 静默跳过 | **已排除** | 4+ 轮 debug log 搜索 "trust"、"skip"、"deny"、"blocked"——零匹配 |
| 路径长度/编码问题 | **已排除** | `C:\t\` 极简目录测试结果一致 |
| `--dangerously-skip-permissions` | **不相关** | 从未使用此标志。Issue #10385 与本环境无关 |

### 4. 第二轮干净验证——正确方法

**测试环境：** 全新 `C:\hook-test2\` 目录，干净的 `.claude\settings.json`，exec form 直接调用 hook binary（不用 cmd.exe，不用 %TEMP%）。

**测试命令：** `claude -p list files`（不加 `--debug-file`，不加任何特殊标志）

**结果（两次独立运行）：**
```
claude_421e3232: hook=Stop, state=done, session_source=payload  ← 第1次
claude_97ffd79d: hook=Stop, state=done, session_source=payload  ← 第2次
```

**结论：settings.json hooks 在首次启动时正确加载执行。** 之前的相反结论是假阴性——由 `cmd.exe + %TEMP%` 测试方法导致。

## 问题诊断与决策过程

### 问题 1：Hooks 不写入 state.json（原始调查）

**描述：** 原始调查者报告 `.claude/settings.local.json` 中的 hooks 从不写入 state.json，尽管尝试了多种格式。

**第一轮调查：**
- 确认 hook binary 在隔离环境中正常工作（`cmd /c type payload | hook.exe`）
- 确认 Codex hooks 正常工作（`commandWindows` 字段）
- 修改 understand-anything plugin hooks.json → hooks 注册并执行
- 构建了"settings.json hooks 启动时不加载"的理论

**第二轮调查（本次会话）：**
- 发现 `%TEMP%` 环境变量在 exec form 上下文中不可用
- 直接用 hook binary 重测（不用 cmd.exe）→ hooks 正常执行
- 部署工作 `.claude/settings.json`

**根本原因（修正后）：** 原始调查的 marker 文件测试（`cmd.exe /c echo > %TEMP%\file`）不可靠，因为 `%TEMP%` 在 Node.js `spawn()` exec form 上下文中不可用。

### 问题 2：误导性的 Debug Log

**问题：** `Hooks: Found 0 total hooks in registry` 持续出现，但 hooks 确实会执行。

**假设：** 此计数器可能只统计 plugin 注册的 hooks，不包含 settings 文件中的 hooks。hooks event registry（代码中的 `Lwy`）可能区分了 settings hooks 和 plugin hooks。

**状态：** 未完全解释，但经实证确认不影响（hooks 正常工作）。

### 问题 3：本地 Plugin Marketplace 解析失败

**尝试：** 在 `%USERPROFILE%\.claude\plugins\cache\cc-traffic-light-hooks\latest\` 创建独立的 hooks-only plugin。

**结果：** `Plugin not available for MCP: cc-traffic-light-hooks@local - error type: marketplace-not-found`

**状态：** 已确认——Claude Code v2.1.204 需要 marketplace 解析才能加载 plugin。本版本不支持纯本地 plugin 注册。

**Workaround：** 不需要——settings.json hooks 在不使用 plugin 的情况下正常工作。

### 考虑的方案

| 方案 | 状态 | 优点 | 缺点 |
|---|---|---|---|
| `.claude/settings.json`（exec form） | **已部署** | 标准方法、可共享 | 路径硬编码为 debug 构建 |
| 修改 UA plugin hooks.json | 不需要 | 可行 | plugin 更新会被覆盖 |
| 创建本地 plugin | 受阻 | 干净隔离 | 需要 marketplace 解析 |
| HTTP hooks | 未尝试 | 无 shell 问题 | 需要本地 HTTP server |

### 最终决策

使用 `.claude/settings.json` + exec form（`args: []`）+ 绝对路径。这是标准文档方法，经验证正常工作。

## 关键决策与上下文

### 正常工作的方式

- **配置文件：** `.claude/settings.json`
- **Hook 形式：** Exec form（`"args": ["claude", "<EventName>"]`）
- **命令路径：** Windows 绝对路径，反斜杠转义
- **事件覆盖：** 6 个事件（UserPromptSubmit, PreToolUse, PostToolUse, Notification, PostToolUseFailure, Stop）

### 不工作的方式

- **Shell form**（无 `args` 字段）——Git Bash 路径转义问题，stdin 不传递
- **`${CLAUDE_PROJECT_DIR}`**——无法解析或解析不正确
- **`settings.local.json`**——功能上与 `settings.json` 相同，没有加载优势
- **本地 plugins（`@local` marketplace）**——需要 marketplace 解析
- **`cmd.exe /c` 使用 `%TEMP%`**——exec form 上下文中环境变量不可用

### 关键教训

1. **测试方法和被测系统同样重要。** `cmd.exe + %TEMP%` marker 方法是所有假阴性的根源。始终独立验证测试工具。
2. **`Hooks: Found 0 total hooks in registry` 是一条误导性的 debug log。** 不要用它作为 hooks 未触发的证据。
3. **不要过度解读 minified 二进制中的函数名。** Bun bundles 压缩/重命名函数——字符串匹配不可靠。
4. **`--debug-file` 标志本身可能影响行为。** 不使用它的测试产生了不同的可观察结果。

### 架构上下文

```
Claude Code
  │ (settings.json 中的 hooks 配置)
  ▼
taskbar_widget_hook.exe claude <HookEvent>
  │ (stdin: JSON payload，含 session_id, hook_event_name)
  ▼
state.json  ← %APPDATA%\CcTrafficLight\state.json
  │ (每秒轮询)
  ▼
taskbar-widget.exe → 任务栏红绿灯显示
```

Hook binary 通过 `taskbar-widget/src/bin/taskbar_widget_hook.rs` 的 `handle_hook()` 处理所有事件：
1. 读取 stdin → 解析 JSON
2. 从 payload 提取 `session_id`（如果没有则回退到 "unknown"）
3. 根据 hook 名称确定 `AgentState`（`hook_rules::infer_state`）
4. 通过 `agent_state::apply_hook_event` 写入 state.json
