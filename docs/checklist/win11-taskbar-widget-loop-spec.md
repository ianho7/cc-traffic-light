# Win11 Taskbar Widget Loop Spec

## Goal

- 交付一个面向当前主力 Win11 机器的独立 Rust PoC
- 外部可观察结果是：任务栏中出现一个固定文本模块，关闭后消失，重启后可复现
- 进展证据不是“代码写了多少”，而是阶段验证信号是否从普通窗口推进到探测、嵌入、定位、复现

## State

源事实：

- 当前目标系统信息：Win11 版本、任务栏对齐方式、缩放比例
- 当前实现阶段：Phase 1 到 Phase 6
- 当前 Win11 真实窗口路径：根窗口、父窗口、锚点窗口
- 最近一次验证结果：`cargo check`、`cargo run`、手动观察结论
- 最近一次失败分类：探测失败、`SetParent` 失败、定位失败、生命周期失败

必须持久化的信息：

- 选定的 Win11 单路径决策
- 当前最小状态结构
- 最后一次可工作的坐标与尺寸
- 关键错误码和句柄信息
- 未完成 checklist 项与阻塞原因

可丢弃的信息：

- 冗长调试过程中的中间猜测
- 已被后续证据推翻的定位尝试

## Planner

下一步选择规则：

- 永远选择“最小且能改变验证状态”的任务
- 优先修复当前阶段的失败，不跳到后续阶段靠新增逻辑掩盖问题
- 如果某阶段的完成信号还没出现，不允许进入下一阶段

触发重规划的条件：

- 当前主力 Win11 真实窗口结构与预期不一致
- `SetParent` 成功但视觉结果与层级判断矛盾
- 重复启动结果不一致
- 为通过某一阶段而不得不引入超出 MVP 范围的抽象

## Actor

允许动作：

- 修改独立 Rust PoC 的最小源文件
- 运行 `cargo check`、`cargo run`
- 添加最小 `println!` 调试输出
- 调整句柄探测、嵌入调用、定位参数

受限动作：

- 不引入 GUI 框架
- 不引入插件系统、配置系统、主题系统
- 不同时实现 Win10 与 Win11 两套路径
- 不为未来扩展提前建立策略框架

## Observer

每轮必须收集的原始输出：

- 编译结果
- 控制台日志中的句柄、错误码、矩形、坐标
- 人工观察结论：普通窗口 / 浮窗 / 任务栏模块 / 不可见
- 已完成 task id 与对应反射文档路径

证据存放：

- 任务反射保存到 `docs/reflections/task-<task-id>-<timestamp>.md`
- Win11 路径与定位规则总结回写到项目文档

## Verifier

验证顺序：

1. `cargo check`
2. 普通窗口基线验证
3. 任务栏句柄探测验证
4. `SetParent` 结果验证
5. 任务栏内可见性验证
6. 重复启动一致性验证

独立完成标准：

- 不能只靠“代码逻辑看起来没问题”判定成功
- 必须同时满足日志信号与肉眼观察信号

## Failure Semantics

失败分类与动作：

- 瞬时失败：仅限偶发编译或运行问题，可重试一次
- 策略失败：父窗口选错、锚点选错、坐标策略错误，必须改策略后再试
- 环境失败：当前 Win11 结构与假设差异过大，先记录真实结构，再改 checklist 对应项
- 范围失败：为了推进当前阶段开始引入透明、多屏、双策略，立即回退
- 未知失败：连续两次无法解释时，暂停编码，先补观测与日志

重试上限：

- 同一失败原因最多直接重试 1 次
- 第二次失败后必须改变假设、策略或观测方式

## Exit Conditions

成功退出：

- checklist 的 Completion Criteria 全部满足

阻塞退出：

- 当前机器的 Win11 路径无法通过现有观测确定
- 没有足够证据判断父窗口或锚点应该是谁
- 下一步只能依赖外部环境变化或人工决策

预算退出：

- 连续多轮都只在微调坐标但没有新增证据
- 为维持循环而开始做低收益试错

风险退出：

- 下一步需要越过当前 MVP 边界，例如同时展开多系统兼容

人工接管退出：

- 需要用户确认是否接受新的 Win11 窗口路径假设
- 需要用户确认是否更换目标环境

## Policy

- 只允许当前主力 Win11 单路径
- 只允许最小 Win32 + GDI + 子窗口嵌入方案
- 只允许 3 到 5 个核心源文件级别复杂度
- 任何新增抽象都必须能直接帮助当前阶段通过验证

## Risk Register

- 风险 1：Win11 任务栏内部窗口结构与参考假设不一致
- 风险 2：Rust Win32 回调与状态持有方式导致生命周期问题
- 风险 3：窗口已嵌入但坐标错误，肉眼误判为失败
- 风险 4：过早引入双策略或产品级架构导致范围失控

对应缓解：

- 只锁定当前主力 Win11
- 先保证普通窗口基线稳定
- 先用醒目背景与保守坐标
- 每阶段设置停止规则，不通过不下钻

## Goal Recommendation

建议在真正开始编码时启用 `/goal`。

原因：

- 这是一个明显的多阶段运行时闭环任务
- 验证依赖本机 Win11 环境与多轮手动观察
- 需要显式区分 `complete`、`blocked`、`replan`
- 很容易因为定位问题反复空转，没有 loop spec 容易变成无边界试错
